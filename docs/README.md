# 📝→🎨 Kotoe（言絵）

> **「言葉だけで、その画像をどこまで再現できる？」**
> アプリ名は「**言葉（こと）＋絵（え）**」＝**言葉から絵へ**の変換そのものを表す。
> 投稿された画像をユーザーが言語化（テキストで描写）し、その説明文だけを頼りに AI が画像を生成。
> 元画像をどれだけ再現できたかを競う、**言語化スキルのゲーム兼検証アプリ**です。

---

## 🎯 コンセプト

「言葉による画像の描写は、どこまで正確に元画像を再現できるのか？」を体験・検証できるアプリ。

1. ユーザーが **お題画像** を投稿する
2. 別のユーザー（または投稿者）が、その画像を **言葉だけで描写** する
3. その描写文を AI 画像生成に渡し、**再現画像** を生成する
4. 元画像と再現画像を並べ、**再現度** を評価する
   - 他ユーザーによる **再現いいね**（主観評価）
   - **自動類似度スコア**（CLIP 等による客観評価）※拡張
5. 再現度ランキングで、**もっとも言語化が上手いユーザー** を可視化する

> ✅ **採用する設計：ユーザー（人間）が描写を書く版。** ゲーム性（ランキング・再現いいね）はこの設計が前提です。
> AI が自動で描写する案は不採用（観察ツール寄りになりゲーム性が失われるため）。ただし「人間 vs AI 対決」として一部取り込む拡張は検討（後述）。

## 💡 背景・着想

2つの問題意識を結びつけたところから着想しています。

1. **「日常にあるものを、言葉で正しく表現するとはどういうことか」** という普遍的な問い。私たちは見たものを毎日言葉にしているが、その描写が本当に正確かを確かめる機会は少ない。
2. **生成AIのプロンプトを実務で書く人が増えている** という現実。「意図した画像を言葉で引き出す力」は、いまや具体的な職能になりつつある。

この2つは「**画像を言葉でどれだけ正確に表現できるか**」という一点で結びつく。本サービスは、その描写力を **AIによる再現** で可視化し、楽しみながら鍛えられる場として実現することを目指す。単なるゲームではなく、**実務で価値を持つ「言語化／プロンプト記述」スキルの訓練・計測**という側面を持たせる。

## 🔍 先行事例と差別化

### 近い先行事例
- **ArtWhisperer**（スタンフォードの研究ゲーム）：目標画像に近づくプロンプトを反復的に探す仕組み。コアは近いが、**運営が用意した固定画像**を使う研究用のデータ収集が目的。
- **「プロンプト当て」系**（Find The Prompt / Unprompted 等）：AI画像を見て元のプロンプトを当てる、日替わりパズル。**向きが逆**で画像生成の工程がない。
- **image-to-prompt ツール**：画像から復元用プロンプトを逆算する実用ツール。ゲーム性はない。

### 差別化ポイント
- **ユーザー投稿型（UGC）**：お題画像をユーザー自身が投稿。先行事例の固定画像と異なり、コンテンツがコミュニティ駆動で増え続ける双方向プラットフォーム。
- **ソーシャル/競争レイヤー**：ランキング・再現いいね・お気に入りで「**描写が上手い人**」を可視化。先行事例にはない。
- **「言語化スキル」のフレーミング**：プロンプト工学ではなく「言葉で正確に描写する力」に焦点。実務（生成AIのプロンプト記述）にも通じる位置づけ。
- **日本語圏**：先行事例はほぼ英語。日本語で言語化を競う場は空いている。
- **差別化を設計で尖らせる案**：「1回の描写で勝負（反復しない）」「モデル向けキーワード禁止・自然文のみ」といった制約で、プロンプト芸ではなく純粋な描写力のゲームにできる。

## ✨ 主な機能

### 認証
- アカウント新規作成 / ログイン / ログアウト
- マイページ（自分の投稿・挑戦・**下書き（保存したプロンプト）**・お気に入り一覧）

### コア機能
- **画像投稿機能**：お題となる画像をアップロード（Cloudinary 保存）
- **言語化（描写）機能**：お題画像に対してテキストで描写を入力。「保存」（下書き）と「画像を生成」の2ボタンを用意。**「保存」したプロンプトは下書き（status: draft）として残り、マイページの「下書き」タブから後で確認・編集・生成できる**
- **画像生成機能**：描写文を画像生成 API に渡して再現画像を生成（**非同期処理**）。**生成したら即公開**（プロンプト入力の実感を重視し、結果を見てから選ぶことはできない）
- **削除機能（論理削除）**：生成結果が気に入らなければ削除可能。物理削除ではなく `discarded_at` フラグによるソフトデリート（再現いいね・通報の参照を壊さない／モデレーション記録を残すため）。**ただし削除しても、その生成は1日の生成回数上限から戻さない**（無限リトライによるチェリーピッキングと API コスト暴発の防止）
- **比較ビュー**：元画像と再現画像を並列表示
- **再現いいね機能**：再現画像（=描写）の出来に他ユーザーが投票
- **お気に入り機能**：気に入った投稿/再現をストック

### 一覧・回遊
- **ランキング機能（全体）**：再現いいね数 / 類似度スコアによる、ユーザー・挑戦の全体ランキング
- **ベスト再現ランキング（お題ごと）**：各お題の挑戦を再現度順に並べ、上位（例：トップ3）を「このお題のベスト再現」として強調。MVP は**いいね数順**、CLIP 導入後は**類似度スコア順**に切り替え可能（「人気順／再現度順」の選択式）
- **検索機能**（ransack）
- **ページネーション**（kaminari）

### 運用・安全性（公開アプリとして必須）
- **コンテンツモデレーション**：投稿画像・生成画像の不適切コンテンツ検出
- **通報機能**：ユーザーによる不適切投稿の報告
- **生成回数制限（レートリミット）**：API コスト暴発の防止（ログインユーザー限定 + 1日あたり上限）

### あると良い拡張
- SNS シェア（X など）— ゲーム性と相性が良い
- 自動類似度スコア（CLIP 埋め込みのコサイン類似度）
- コメント機能

### 画像生成 API の選定（重要）

> ⚠️ **DALL·E 2 / 3 は 2026年5月に API から削除済み**のため選択不可。

| 候補 | 概算コスト | 特徴 |
|---|---|---|
| **OpenAI GPT Image 1 Mini** | $0.005〜/枚（Low品質） | **導入が最もラク・プロンプト追従性◎。個人開発の入口に推奨** |
| OpenAI GPT Image 2 | $0.005〜$0.21/枚 | 最新フラッグシップ。高品質だが高価 |
| Google Imagen 4 Fast | $0.02/枚 | 価格性能比◎。ただし Vertex AI のセットアップが必要 |
| Flux（Replicate / fal.ai 経由） | $0.008〜$0.04/枚 | 安価。ホスト型アグリゲーター経由 |

→ **まずは GPT Image 1 Mini の Low 品質で開発・検証し、必要に応じて品質を上げる**のがコスト効率的。

> 🔒 API キーは必ず **Rails サーバー側**に保持し、フロントエンドには絶対に出さないこと。

### 非同期処理の選定：Solid Queue（決定済み）

Sidekiq と比較した結果、**Solid Queue を採用する**（導入は issue 4-1）。

| | Sidekiq | Solid Queue |
|---|---|---|
| ジョブの保存先 | Redis（別インフラが必要） | PostgreSQL（追加インフラ不要） |
| 取得方式 | ブロッキング（即時） | ポーリング（〜1秒の遅延） |
| Render での実行 | **独立プロセスが前提 → 有料のワーカーサービスが必要** | Puma のプラグインとして Web プロセス内で動く → **無料枠に収まる** |

決め手は **Render の無料プランではバックグラウンドワーカーが別サービス扱いで有料になる**こと。Solid Queue なら Web サービス1つで完結し、Redis も不要。

画像生成は1回10〜60秒かかる長いジョブで、しかも1日の生成回数に上限をかけるため、Sidekiq の「即時取得」という強みは活きない（Solid Queue のポーリング遅延1秒は体感に出ない。フロントは元々ポーリングで完了を検知する設計）。

> ⚠️ 懸念：Solid Queue はポーリング型なので、ワーカーが生きている間 Neon にクエリを投げ続け、Neon の自動サスペンドを妨げうる。ただし Render の無料 Web サービスは15分アイドルでスリープするため、夜間はワーカーごと止まり実害は小さいと判断。実測は issue 4-1 / 8-2 で行う。

## 🏗 アーキテクチャ

```
[ User ]
   │
   ▼
[ Next.js (Vercel) ]  ──── JWT ────▶  [ Rails API (Render) ]
                                          │
                          ┌───────────────┼────────────────┐
                          ▼               ▼                ▼
                   [ PostgreSQL ]   [ Cloudinary ]   [ 非同期ジョブ ]
                   (Neon)           (画像保存)        (Solid Queue)
                                                           │
                                                           ▼
                                                  [ 画像生成 API ]
```

### ⏱ 画像生成の非同期フロー（必須）

画像生成は数秒〜数十秒かかるため、**リクエストを同期で待たせない**設計が必須です。

```
1. ユーザーが「画像を生成」を押す（描写文は先に保存される）
2. Rails が Attempt(status: generating) を作成し、GenerateImageJob を enqueue
   ※生成回数（1日の上限）はこの時点で消費する
3. ジョブが画像生成 API を呼び出し → Cloudinary にアップロード
   → (拡張) CLIP 類似度を計算 → Attempt を更新(status: published) ＝ 即公開
   → 失敗時は status: failed（ユーザーは再試行可能）
4. フロントエンドがポーリング / SSE で完了を検知し、再現画像とスコアを表示
```

> ⚠️ Render の無料プランはアイドル時にスリープするため、**コールドスタートの遅延**を考慮すること。

## 🗃 データモデル案（人間が描写する設計の場合）

```
User
  ├─ has_many :posts
  ├─ has_many :attempts
  ├─ has_many :likes
  └─ has_many :favorites

Post（お題画像）
  ├─ belongs_to :user
  ├─ image (Cloudinary)
  ├─ title
  └─ has_many :attempts

Attempt（再現挑戦）
  ├─ belongs_to :post
  ├─ belongs_to :user
  ├─ description       # ユーザーが書いた言語化テキスト
  ├─ generated_image   # 生成された再現画像 (Cloudinary)
  ├─ similarity_score  # (拡張) CLIP 類似度などの自動スコア
  ├─ status            # draft / generating / published(=完了・即公開) / failed
  ├─ discarded_at      # 論理削除（ソフトデリート）用のフラグ
  └─ has_many :likes

Like（再現いいね）
  ├─ belongs_to :attempt
  └─ belongs_to :user

Favorite（お気に入り）
  ├─ belongs_to :user
  └─ belongs_to :post  (または :attempt)
```

> ランキングは Like 数 / similarity_score を集計して算出（`status: published` かつ `discarded_at` が nil のものが対象）。

## 🧭 技術選定の理由（Next.js + Rails API）

本プロジェクトは Rails モノリス + Hotwire でも実装可能で、認証・非同期処理はむしろそちらの方が簡潔です。それを理解した上で、以下の目的から **あえて分離構成（Next.js + Rails API）** を選択しています。

- **end-to-end のオーナーシップを示す**：フロント・バック・デプロイまで一人で通して構築できることを示す（キャリアチェンジの文脈で有効）。
- **比較ビューの UX**：元画像と再現画像を並べる/重ねる体験など、クライアント状態を持つリッチな UI は React/Next.js が作りやすい。
- **React/Next.js の実務スキル獲得**：市場価値の高いフロント技術を実装で身につける。

### 引き受けたトレードオフ（実装で対処する課題）
- **別ドメイン間の認証**：Vercel ↔ Render はクロスオリジンのため、JWT を Authorization ヘッダで送る方式で対処（cookie 共有は使わない）。
- **非同期生成の完了検知**：Hotwire の Turbo Streams のような自動プッシュが無いため、ポーリング / SSE をフロントで実装。
- **その他**：CORS 設定、デプロイ2系統、フロント/バックの型の二重管理。

> 💡 「簡単な道があると知った上で、目的のために難所を選び、その代償を自分で処理した」——この検討・選定・対処の過程そのものが、面接での技術選定アピールの核になる。
