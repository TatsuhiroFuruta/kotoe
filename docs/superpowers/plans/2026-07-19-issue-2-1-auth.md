# issue 2-1 devise + devise-jwt による認証API Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** JWT を Authorization ヘッダで運ぶ認証APIを作り、`POST /api/auth/sign_up` / `sign_in`・`DELETE /api/auth/sign_out`・`GET /api/me` が動き、sign_out 後は同じトークンが弾かれる状態にする。

**Architecture:** devise を最小構成（`database_authenticatable` / `registerable` / `validatable` / `jwt_authenticatable`）で導入する。トークンの失効は devise-jwt の Denylist 戦略で、sign_out したトークンの `jti` を `jwt_denylist` テーブルに記録して以降拒否する。API モードなので Devise の HTML リダイレクト動作は切り、コントローラは `respond_with` を上書きして JSON を返す。認証失敗（パスワード不一致・トークン無し）はコントローラに到達せず Warden が処理するため、`Devise::FailureApp` を継承したカスタム failure app で JSON を返す。

**Tech Stack:** Ruby 3.4.10 / Rails 8.1（API モード）/ PostgreSQL / devise / devise-jwt / RSpec + FactoryBot。開発・テストは docker compose 上で実行。

## Global Constraints

- 文字列は**ダブルクォート**（モデル・コントローラ・spec・factory すべて）。rubocop は `rubocop-rails-omakase` ベース。`db/schema.rb` と `db/migrate/**` は rubocop 対象外。
- **1 issue = 1 ブランチ = 1 PR**。ブランチは `feature/issue-2-1-auth`（作成済み）。main へ直接コミットしない。
- コミット前に `docker compose exec backend bundle exec rubocop` と `docker compose exec backend bundle exec rspec` を通す。
- コマンドはすべて起動中の docker compose で実行する（先に `docker compose up -d` しておく）。
- FactoryBot は `create(...)` / `build(...)` で呼べる設定済み（`spec/rails_helper.rb`）。`spec/support/**/*.rb` は `rails_helper` が自動で読み込む。spec の type は置き場所（`spec/models` / `spec/requests`）から自動推論される。
- **エラー文言はバックに持たせない**。Rails はエラーコード（`invalid_credentials` / `unauthorized`）とフィールド別のエラーキーを返し、日本語の文言はフロントが持つ（CLAUDE.md「メッセージ・エラー・i18n の責務」）。
- **秘密情報をリポジトリに置かない**。`JWT_SECRET_KEY` の実値は `.env.development`（git 管理外）と CI の env にだけ書く。
- この issue では `config/initializers/cors.rb`・フロントエンド・本番環境設定に**触らない**（それぞれ 2-2 / 7-1 / 8-2a の担当）。
- 422 のステータスシンボルは Rails 8.1 の `:unprocessable_content` を使う（`:unprocessable_entity` は非推奨）。
- コミットメッセージ末尾に `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>` を付ける。

JWT を Authorization ヘッダで運ぶ認証APIを追加した。設計は `docs/superpowers/specs/2026-07-19-issue-2-1-auth-design.md`。

## 追加したエンドポイント
| メソッド | パス | 役割 |
|---|---|---|
| POST | `/api/auth/sign_up` | 新規登録（201・JWT 発行） |
| POST | `/api/auth/sign_in` | ログイン（200・JWT 発行） |

| GET | `/api/me` | ログイン中のユーザー情報 |

## 設計判断
- devise は**最小構成**（`database_authenticatable` / `registerable` / `validatable` / `jwt_authenticatable`）。パスワードリセット・メール確認はメール送信基盤が要るため MVP では持たない。
- 失効は **Denylist 戦略**（`jwt_denylist` テーブル）。ログアウトしたトークンを即座に無効化できる。
- JWT の有効期限は **24時間**。
