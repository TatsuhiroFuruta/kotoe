## 1. サービス概要（最後に記入）

🟦【下書き／最後に見直す】
- Kotoe（言絵）は、**画像を「言葉だけ」で描写し、その言葉から AI が画像を再現して、再現度を競う**Webサービス。
- **画像生成AIのプロンプトを書く力／物事を言葉で正確に伝える力**を鍛えたい人が、ゲーム感覚で楽しみながら、自分の「言語化の精度」を可視化して上達できる。
- 「言葉から絵へ（こと＋え）」がそのまま名前になっている。

## 4. 想定ユーザー

### 4-1. 想定しているユーザー（顔が浮かぶ1人）

🟦【下書き：方向性「プロンプトを上達させたい人／言葉で伝える力を鍛えたい人」を、1人に絞った案】
> 採用担当に深掘りされても崩れないよう、**1人**に絞っています。違和感があれば、あなたが実際に顔を思い浮かべられる人に差し替えてください。

**ペルソナ（主）**：最近、仕事で画像生成AIを使い始めた20代後半〜30代前半の若手（Webデザイナー／制作・広報・マーケ職など）。

- 年齢：26〜32歳
  - 理由：画像生成AIを**業務で実際に触り始めている層**で、「プロンプト＝仕事のスキル」という実感を持ち始めている。学生ほど時間は無いが、上達意欲とツールへの抵抗の少なさが両立する年代。
- 生活・状況：イメージは頭にあるのに、プロンプトで思い通りの画像が出せず、何度も書き直して時間を溶かしている。「うまく言葉にできない」のがボトルネックだと薄々感じている。

- 環境：スマホ／PCの空き時間（通勤・休憩・寝る前）にスキマでできる。
  - 理由：1問が短時間で完結する設計（描写→生成→結果）なので、まとまった時間が取れなくても続けられる。

**ペルソナ（副・隣接層）**：文章で伝える力を鍛えたいライター志望・就活生など。主ペルソナで設計し、副は「広がる先」として意識する程度に。

## 5. 既存サービス・競合調査

### 5-1. 似たサービスの調査

🟦【下書き】
- サービス名：**ArtWhisperer**（スタンフォードの研究プロジェクト）
  - URL：https://arxiv.org/abs/2306.08141 ／ https://github.com/kailas-v/ArtWhisperer
  - できること：目標画像を見せ、それに近い画像を生成するプロンプトを**反復的に**探させるオンラインゲーム。5万件超の人間とAIのやり取りを収集し、生成画像と目標画像の**類似度を自動スコア化**して提示する。あくまで研究用のデータ収集が目的。
- （参考）**「プロンプト当て」系**（Find The Prompt / Unprompted など）：AI画像を見て元のプロンプトを当てる日替わりパズル。Kotoe とは**方向が逆**で画像生成の工程がない。

### 5-2. それでも自分が作りたい理由（不満・ズレ）

✍️【あなたが書く欄／下に下書き。自分の言葉に】
🟦 下書き：ArtWhisperer は「運営が用意した固定画像」を「英語で」「研究のために」扱うもので、**自分が日常で出会った“このモノ”を言葉にして練習する**という体験ではない。プロンプト当て系は「正解を当てる」ゲームで、**自分の描写力そのものを鍛える**ものではない。「言葉で正確に伝える力を、自分の題材で楽しく鍛えたい」という欲求に応えるものが見当たらなかった。
→ あなたの違和感を一言で：＿＿＿＿＿＿

### 5-3. 差別化を「体験価値」で表現する

🟦【下書き】
- Kotoe は、**【最近プロンプトで思い通りの画像が出せず悩む若手】の【“言葉にできない”もどかしさを感じた瞬間】を一番助ける**アプリ。
- 「機能がある／ない」ではなく、**「自分の描写が通じたかを“絵”で即フィードバックされ、反復で言語化が上手くなっていく」体験**が核。

## 8. このアプリの懸念点とその対策

🟦【下書き：会話で出た論点から。各項目はあなたの言葉で補強を】

### 懸念点①：UGC の「鶏と卵」問題
- 何が問題になりそうか：お題（投稿）が少ないと挑戦が集まらず、挑戦が無いとお題も増えない。
- なぜそう思うか：投稿型サービスは初期のコンテンツ不足で過疎りやすいため。
- 対策：初期は自分で良質なお題を一定数仕込む／1問が短時間で完結する設計にして挑戦のハードルを下げる。

### 懸念点②：即公開＋削除による“裏口チェリーピッキング”とAPIコスト
- 何が問題になりそうか：生成し直して良い結果だけ残せると、ランキングの公平性が崩れ、コストも膨らむ。
- 対策：削除しても生成回数は戻さない／1日あたりの生成上限を設ける。

### 懸念点③：不適切な投稿・生成
- 何が問題になりそうか：公開UGCのため不適切画像が混じるリスク。
- 対策：投稿・生成画像のモデレーション、通報機能、ソフトデリートでの記録保持。

## 10. 技術スタック（手段としての技術）

### 10-1. 使用予定の技術
🟦【下書き】
- フレームワーク：フロント Next.js（App Router）+ TypeScript + TailwindCSS／バック Ruby on Rails（API）
- DB：PostgreSQL（Neon）
- デプロイ先：フロント Vercel／バック Render
- 使用予定ライブラリ：devise + devise-jwt（認証）、discard（論理削除）、ransack（検索）、kaminari（ページング）、Sidekiq または Solid Queue（非同期）、Cloudinary（画像保存）、画像生成API（OpenAI GPT Image など）

### 10-2. 技術選定の比較検討
🟦【下書き】

**フロント構成**
| 候補 | 採用/不採用 | 理由 |
| --- | --- | --- |
| Next.js + Rails API | 採用 | フロント〜バック〜デプロイまで一人で通す力（end-to-end）を示せる／比較ビューなどリッチUIに強い／React は求人が多い |
| Rails モノリス + Hotwire | 不採用 | 認証も非同期も簡潔で速く作れるが、フロント力のアピールが弱い。今回は学習と幅を優先して見送り |

**認証の運び方**
| 候補 | 採用/不採用 | 理由 |
| --- | --- | --- |
| JWT（Authorizationヘッダ） | 採用 | Vercel↔Render が別ドメインのため、クロスオリジンで素直に動く |
| セッションCookie | 不採用 | 別ドメインだと SameSite/CORS の設定が複雑でハマりやすい |
> ※「OAuth（ソーシャルログイン）」は“ログイン方法”の話で JWT と排他ではない。MVP はメール＋パスワード、OAuth は拡張。

**画像生成API**
| 候補 | 採用/不採用 | 理由 |
| --- | --- | --- |
| OpenAI GPT Image（Mini・低品質から） | 採用（候補筆頭） | 導入が容易・プロンプト追従性が高い・低コストから始められる |
| Google Imagen 4 Fast | 比較対象 | 安価だが Vertex AI のセットアップが必要 |
| DALL·E | 不採用 | 2026年5月にAPIから削除済み |

- 今回チャレンジしたい点：React/Next.js の本格習得、分離構成（CORS・JWT）、非同期ジョブ＋外部API連携。
- 不安な点：✍️【あなたが書く欄】＿＿＿＿（例：React が初挑戦でIssueの見積もりが読みにくい、など）

### 10-3. キャッチアップ不足の懸念
🟦【下書き】
- React/Next.js は今回が本格的な初挑戦のため、**完成までのハードルが高い**こと、**Issueの見積もりが難しい**ことが懸念。
- 対策：①まず Rails 側（モデル・認証・API）と非同期フローを**画像生成はダミー（固定画像）**で先に通し、動く土台を作る。②そのうえで Next.js を段階的に接続。③画像生成APIの本接続やリッチUIは後段に回し、各ステップを小さく区切って見積もりの精度を上げる。
- ✍️ 自分の現状（どこまで分かっていて、何が未知か）を一言：＿＿＿＿
